# 영수증 처리 프로세스 개선 제안

**AI센터 | ETC 영업사원 지출 영수증 처리 자동화**

---

## 1\. 배경 및 현재 프로세스

ETC 영업사원들은 병의원 방문 시 설명회 등 다양한 비용을 지출합니다. 이 영수증을 SFE(영업관리 시스템)에 첨부하면, RPA가 자동으로 OCR 처리 후 DB에 저장하고, 승인자가 검토 후 최종 승인하는 구조입니다.

### 현재 흐름 (AS-IS)

영업사원 → SFE 영수증 업로드

         → RPA 2시간마다 폴링 (봇 2대)

         → SSH로 SFE 서버 접속, 이미지 다운로드

         → 오픈소스 OCR로 텍스트 추출

         → Qwen2.5 7B로 품목 구조화

         → SFE DB 업데이트

         → 승인자가 승인화면에서 확인 후 승인

---

## 2\. 현재 문제점

| 문제 | 내용 |
| :---- | :---- |
| OCR 품질 저하 | 오픈소스 OCR 텍스트 품질 낮아 LLM 결과 부정확 |
| 처리 지연 | 2시간 폴링 방식으로 월말 집중 시 지연 빈번 |
| 동시 처리 한계 | RPA 봇 2대 한계, 다수 건 동시 처리 불가 |
| 사전 차단 불가 | 수기 영수증·주류 포함 여부 자동 감지 기능 없음 |
| CP 규정 검토 없음 | 비용 한도·품목 제한 등 자동 검토 기능 없음 |

---

## 3\. 개선 방향 — 3가지 핵심 과제

### 01\. VLM 직접 분석

- OCR 단계 제거 → Qwen3-VL이 영수증 이미지 직접 판독  
- 텍스트 추출 오류 근본적 해결  
- 품목·금액·장소 정확도 향상  
- 현재 사내 GPU 서버에 Qwen3-VL 설치 완료

### 02\. 사전 차단 로직

- 수기 영수증 자동 감지 → 승인자 도달 전 반려 처리  
- 주류 포함 여부 자동 감지 → 플래그 또는 차단  
- 승인자 검토 부담 사전에 경감  
- 영업사원에게 즉시 재제출 요청

### 03\. CP 규정 자동 검토

- CP 규정(비용 한도, 품목 제한 등)을 프롬프트에 직접 주입  
- 영수증 내용과 규정 위반 소지 자동 판단  
- 판단 근거와 함께 승인자 화면에 표시  
- 규정 변경 시 프롬프트 문서만 교체하면 즉시 반영 (파인튜닝 불필요)

---

## 4\. 개선된 프로세스 (TO-BE)

영업사원 → SFE 영수증 업로드

         → SFE가 미들웨어에 즉시 트리거 (건번호 전달)

         → 미들웨어가 SFE 서버 접속, 이미지 취득

         → Qwen3-VL에 이미지 \+ CP 규정 프롬프트 전달

         → Qwen3-VL이 한번에 분석:

             \- 품목/금액 추출

             \- 수기 영수증 여부 판단

             \- 주류 포함 여부 판단

             \- CP 규정 위반 소지 판단

         → SFE DB 업데이트

         → 문제 있으면 사전 차단 / 없으면 승인자에게 분석 결과와 함께 전달

### AS-IS vs TO-BE 비교

| 항목 | 기존 (AS-IS) | 개선 (TO-BE) |
| :---- | :---- | :---- |
| 실행 방식 | 2시간마다 폴링 | 업로드 즉시 트리거 |
| 동시 처리 | 봇 2대 한계 | 제한 없음 |
| 이미지 분석 | OCR → Qwen2.5 7B | Qwen3-VL 직접 분석 |
| 수기 영수증 | 자동 감지 불가 | 자동 감지 후 차단 |
| 주류 포함 | 정확도 낮음 | VLM 직접 판단 |
| CP 규정 검토 | 없음 | 프롬프트 기반 자동 판단 |
| 월말 처리 지연 | 자주 발생 | 즉시 처리로 해소 |

## 5\. 기술 구성 및 미들웨어 역할

SFE 시스템 (AWS)          미들웨어 서버              사내 GPU 서버

──────────────       ──────────────────────       ──────────────

• 영수증 업로드   →   • SFE에서 이미지 취득    →   • 영수증 이미지 분석

• 트리거 발송        • Qwen3-VL API 호출          • 품목/금액 추출

• 결과 수신 및 표시  • CP 규정 프롬프트 주입       • 수기/주류 판단

                     • 결과 파싱 및 검증           • CP 규정 위반 검토

                     • SFE DB 결과 저장

                     • 에러 처리 / 로깅

### CP 규정 프롬프트 방식

CP 규정 문서(비용 한도, 품목 제한 등)를 미들웨어가 프롬프트에 직접 삽입하여 Qwen3-VL이 영수증과 대조해 위반 소지를 판단합니다.

- **파인튜닝 불필요** — 규정 변경 시 문서만 교체하면 즉시 반영  
- **판단 근거 함께 리턴** — 승인자가 이유를 확인 가능  
- **규정 분량에 따라 RAG 방식으로 고도화** 가능

## 6\. 담당자별 확인 사항

### SFE 담당자

| \# | 확인 항목 | 내용 |
| :---- | :---- | :---- |
| Q1 | 트리거 발송 가능 여부 | SFE에서 영수증 업로드 시 외부 API(미들웨어)로 웹훅/HTTP 호출이 가능한지 |
| Q2 | 네트워크 연결 (AWS ↔ 사내 GPU 서버) | VPN 또는 전용선 경유 API 호출 가능 여부 |
| Q3 | 이미지 전달 방식 | SFE가 직접 전달 가능한지, 불가시 SSH 접속 취득 방식 허용 여부 |
| Q4 | 보안 정책 | 영수증 이미지가 AWS 외부(사내 서버)로 전송되는 것에 대한 보안·개인정보 정책 검토 |
| Q5 | 사전 차단 알림 방법 | 반려 처리 시 영업사원에게 알림을 줄 수 있는 SFE 내 기능 유무 |
| Q6 | SFE DB 구조 공유 | 분석 결과(품목, 수기여부, 주류여부, CP 판단 등)를 저장할 필드 확인 및 추가 필요 여부 |

### CP 담당자

| \# | 확인 항목 | 내용 |
| :---- | :---- | :---- |
| Q1 | CP 규정 문서 형태 | PDF, 엑셀, 내규 문서 등 형태 확인 — 텍스트 추출 가능 여부 |
| Q2 | 규정 분량 | A4 몇 장 분량인지 — 프롬프트 직접 삽입 vs RAG 방식 결정 기준 |
| Q3 | 위반 처리 방침 | 완전 차단인지, 플래그만 달고 승인자에게 넘기는 방식인지 |
| Q4 | AI 판단 책임 소재 | AI 판단이 틀렸을 경우 처리 절차 — 최종 판단권은 사람(승인자)에게 유지 필요 |
| Q5 | 규정 변경 주기 | 프롬프트 업데이트 절차 및 관리 주체 확인 |
| Q6 | 우선 적용 위반 유형 | 주류 포함, 1인당 금액 한도, 품목 제한 등 우선순위 결정 |
