01

배경 및 목적

팀에서 Claude API를 통해 AI 에이전트(OpenClaw)와 RPA(Checkmate) 등을 운용하고 있었으나, 사용량 증가에 따른 토큰 비용 부담이 커지고 있었습니다. 사내에 H100 NVL GPU 서버가 있어 이를 활용한 자체 LLM 서비스 구축을 결정했습니다.

💰
비용 절감
Claude API 토큰 비용을 자체 서버 운영비로 대체. 사용량이 많을수록 ROI 향상.
🔒
데이터 보안
사내 데이터가 외부 API로 전송되지 않아 정보 보안 강화.
🖼️
멀티모달 지원
이미지 캡처 붙여넣기 → 이해하고 대화하는 Vision 기능 필요.
⚡
Sonnet급 성능
Claude Sonnet 수준의 코딩/추론 능력을 자체 서버에서 구현.

양자화 선택: FP16 → FP8

초기에는 원본 FP16(약 65GB)을 사용했으나 GPU 메모리의 87%를 점유하여 다른 서비스에 영향을 줄 수 있었습니다. Qwen 공식 팀이 직접 출시한 FP8 양자화 버전으로 교체하여 GPU 사용률을 52%로 낮췄습니다.

✅
FP8은 Qwen 공식 팀이 출시한 버전으로, 원본 BF16 모델과 성능이 거의 동일하면서 메모리 사용량은 절반 수준입니다. 일반 대화, 코딩, 이미지 분석 용도에서는 체감 차이가 없습니다.

Checkmate 연동 특이사항

Checkmate는 Ollama API 형식만 지원하는 패키징된 솔루션이라 코드 수정이 불가했습니다. Ollama API → OpenAI API 변환 프록시를 직접 개발하여 해결했습니다.

# /data/ollama_proxy.py - Checkmate용 경량 프록시 from fastapi import FastAPI, Request from fastapi.responses import JSONResponse import httpx, uvicorn app = FastAPI() VLLM_URL = "http://localhost:8003/v1" @app.post("/api/chat") async def chat(request: Request): body = await request.json() body["stream"] = False # 스트리밍 비활성화 async with httpx.AsyncClient(timeout=300) as client: r = await client.post(f"{VLLM_URL}/chat/completions", json=body) content = r.json()["choices"][0]["message"]["content"] return JSONResponse({ "model": body["model"], "message": {"role": "assistant", "content": content}, "done": True })

주요 트러블슈팅

🔴 CUDA 드라이버 버전 충돌
원인: 최신 vLLM(0.21.0)이 설치한 PyTorch가 CUDA 12.9 드라이버와 호환되지 않음
✅ 해결: vLLM Docker 공식 이미지 사용으로 버전 의존성 문제 해소
🔴 루트 파티션 디스크 풀(100%)
원인: pip 캐시(13GB) + HuggingFace 모델 다운로드가 / 파티션에 저장됨
✅ 해결: 모델 저장 경로를 /data (4.8TB 파티션)로 변경, pip 캐시 정리
🔴 Docker 컨테이너 내부 DNS 불통
원인: Docker 기본 네트워크에서 huggingface.co 도메인 해석 실패
✅ 해결: --network=host 옵션으로 호스트 네트워크 직접 사용
🔴 Ollama 프로세스 자동 재시작
원인: Ollama가 Docker 컨테이너로 별도 운영 중이어서 kill 후 자동 재시작
✅ 해결: docker update --restart=no ollama로 자동 재시작 비활성화
🔴 Checkmate 스트리밍 응답 미표시
원인: Checkmate가 스트리밍(SSE) 응답을 파싱하지 못하고 빈 화면 표시
✅ 해결: 커스텀 프록시에서 stream=False로 강제 설정하여 일반 JSON 응답으로 변환
🔴 Tool Calling 비활성화 오류
원인: OpenClaw가 tool_choice: "auto"를 전송하나 vLLM에 tool calling 옵션 미설정
✅ 해결: --enable-auto-tool-choice --tool-call-parser hermes 옵션 추가

향후 계획

📊
모니터링 구축
Grafana + Prometheus로 GPU 사용률, 응답 시간, 요청 수 등 대시보드 구축. vLLM의 /metrics 엔드포인트 활용.
🔑
API 인증 강화
현재 내부망은 인증 없이 접근 가능. API Key 기반 인증 추가로 사용자별 접근 제어.
🚀
추가 모델 검토
남은 GPU 메모리(각 44GB)에 코딩 특화 모델 등 추가 서비스 올리는 것 검토.
📱
모바일 접속
Tailscale을 통한 모바일에서 Open WebUI 접속 환경 구축.
💡
현재 GPU 메모리 여유분이 GPU당 약 44GB입니다. 추가 모델을 올리거나 다른 AI 서비스를 운영할 수 있는 충분한 여유가 있습니다.
